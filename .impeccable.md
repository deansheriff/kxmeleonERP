# DotMac ERP — Impeccable Design Context

## Design Context

### Users
- **Primary**: Finance professionals — accountants, bookkeepers, CFOs managing general ledgers, reconciliations, AR/AP aging, tax filings, and financial reporting for Nigerian businesses.
- **Secondary**: HR and operations staff — payroll officers, HR managers, operations leads handling employee lifecycle, leave, attendance, and procurement workflows.
- **Context**: Cross-functional enterprise teams in mid-to-large organizations. Users spend hours daily in this tool processing transactions, reviewing reports, and making approval decisions. Speed and accuracy are existential — a misposted journal or missed reconciliation has real financial consequences.
- **Job to be done**: "Give me a clear, trustworthy picture of my organization's financial and operational state, and let me act on it fast."

### Brand Personality
- **Three words**: Professional, Trustworthy, Clear
- **Voice**: Authoritative but approachable. Like a senior accountant who explains things plainly — never patronizing, never vague.
- **Tone**: Calm confidence. The interface communicates "everything is under control" through clear hierarchy, predictable patterns, and zero ambiguity.
- **Emotional goal**: **Confidence and control** — "I know exactly where things stand." Users should feel total oversight with no surprises.

### Aesthetic Direction
- **Visual tone**: Modern professional — clean, structured, and precise without being cold or sterile. Warm parchment backgrounds and teal accents give it character without being playful.
- **Identity**: Distinctly a *finance tool* — tabular numbers, ledger patterns, double-entry displays, and aging bars signal domain fluency. Module color-coding (teal for GL, emerald for AR, rose for AP) borrows from the physical colored-tab ledger tradition.
- **Typography**: Serif display font (Fraunces) for page titles adds gravitas; DM Sans body text stays clean and readable; JetBrains Mono for all financial figures ensures precision alignment.
- **Dark mode**: Full support, used by power users in extended sessions. Slate-900 base, never pure black.
- **References**: The polish of Stripe's dashboard, the structure of Xero's accounting UI, the clarity of Linear's information hierarchy.
- **Anti-references**:
  - NOT legacy enterprise (SAP/Oracle desktop-era density, gray everything, overwhelming toolbars)
  - NOT generic SaaS (cookie-cutter Bootstrap/Material UI that looks like every other tool)
  - The corridor is narrow: serious enough for an audit partner to trust, modern enough that a new hire doesn't dread opening it.

### Design Principles

1. **Clarity over cleverness** — Every element earns its place. Financial data must be instantly scannable: right-aligned numbers, status badges with text+color, empty states instead of blank voids. If a user has to think about what they're looking at, the design failed.

2. **Confidence through structure** — Predictable layouts, consistent patterns, visible system state. Dashboards lead with KPIs. Lists always have search, filters, and bulk actions. Detail pages show workflow progress. The user always knows where they are and what they can do.

3. **Warmth without whimsy** — Parchment backgrounds, teal-and-gold accents, self-hosted serif headings, and subtle ledger-line textures give the interface personality and cultural grounding. But animations stay under 400ms, illustrations serve function (empty states, onboarding), and nothing bounces or sparkles.

4. **Accessible by default** — WCAG 2.2 AA minimum. Color is never the sole indicator. Touch targets are 44px. Focus states are visible. Status badges always pair color with text. This isn't optional — it's a regulated-environment requirement.

5. **Speed is a feature** — 300ms debounced search, HTMX partial updates, skeleton loading states, no full-page reloads for list operations. Power users process hundreds of transactions daily — every saved second compounds.

### Tech Stack Context
- **CSS**: Tailwind CSS 3.4 with custom design tokens (CSS variables)
- **Interactivity**: Alpine.js 3.15 + htmx 2.0 (no React/Vue)
- **Templating**: Jinja2 (server-side rendered)
- **Charts**: Chart.js 4.5 + custom SVG sparklines
- **Fonts**: Self-hosted DM Sans, Fraunces, JetBrains Mono (variable, woff2)
- **Components**: 50+ Jinja2 macros in `templates/components/macros.html`
- **Module theming**: Each module has its own base template with colored sidebar (Finance=teal, People=violet, Expense=amber, Inventory=emerald, etc.)

### Key Design Constraints
- All financial values use `font-mono tabular-nums text-right` — no exceptions
- Negative amounts use accounting parentheses `(1,234.56)` in rose, never minus signs
- Minimum text size is 12px (`text-xs`) — never smaller
- Every `<th>` needs `scope="col"`, every icon-only button needs `aria-label`
- Alpine.js `x-data` attributes must use single quotes when containing `tojson`
- Status badges come from the `status_badge()` macro — never inline HTML
- CSRF token required in every POST form
- `| safe` is only allowed for CSRF tokens, `tojson` for JS, and admin-configured CSS
